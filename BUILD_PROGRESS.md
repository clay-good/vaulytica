The coverage test exercises ~40 of the 68 rules with focused positive inputs — the remaining 28 (mostly informational surfacers like RISK-002 asymmetry detection or DARK-002 buried-notice detection) are covered indirectly by the end-to-end determinism test in `all-rules.test.ts` and will gain bespoke tests as fixture contracts arrive.
- **Step 16:** The four fixtures are synthetic; the launch-corpus list in spec §26 step 16 also calls for real Common Paper templates, the GitHub Balanced Employee IP Agreement, an open-licensed commercial lease, and a scanned-PDF variant of the clean NDA. Vendoring those requires network access (GitHub clones) and a print-to-image step (the scanned PDF), so they're tracked in `tests/fixtures/README.md` and the golden test will pick them up automatically once they land in `contracts/`. The runner-determinism fix (`elapsed_ms` blanked when hashing) is load-bearing for the regression check — without it, every fixture run on a different machine would have produced a different `result_hash` and the spec §17 determinism guarantee would have been false. If you bump the engine, re-run `npm run fixtures:regen-golden`, review the diff in `tests/fixtures/expected/`, and commit deliberately — never auto-accept.
- **Step 14:** The deploy workflow expects `CLOUDFLARE_API_TOKEN` and `CLOUDFLARE_ACCOUNT_ID` secrets — those must be set on the repo before the first deploy. The Pages project itself (custom domain `vaulytica.com`, HTTPS enforced, "Always Use HTTPS") is configured in Cloudflare's UI, not via Wrangler; spec §26 step 14 calls that out. `wasm-unsafe-eval` is in `script-src` because pdfjs-dist and tesseract.js compile WebAssembly at runtime — strictly speaking the spec's CSP omits this, but the alternative is the analysis pipeline refusing to instantiate the OCR worker. Use `securityheaders.com` against the live deploy to confirm an A rating once the domain is live. The 500 KB chunk warning from the build is an optimization opportunity for Step 15/16 — pdfjs and tesseract are heavy, and dynamic-importing the OCR fallback is the lowest-hanging fruit. Playwright browsers are only installed in CI (`npm run e2e:install`) to keep local installs slim; running `npm run e2e` locally needs that step too.
- **Step 13:** The service worker scope is the entire origin (`/`). On Cloudflare Pages, the `/sw.js` URL is what the browser uses; on dev it's the same. The PWA install requirements (Chrome Lighthouse) need a 192+ and 512+ PNG and a maskable variant — all three are now committed. Step 14's deploy step copies `site/icon-*.png` + `manifest.webmanifest` to `dist/` automatically via Vite (anything outside `index.html`'s `<script>`/`<link>` graph that lives directly in `site/` is treated as a static asset and copied). If a future Vite upgrade stops auto-copying root-level PNGs, add them to `publicDir`. The vm-based service-worker tests stub `caches`/`self`/`fetch` minimally — fidelity is "the strategy branches correctly," not "Workbox-grade correctness." The DKB stale-while-revalidate intentionally keeps serving old manifests; the surface contract is "the next page load picks up the new DKB" rather than "this load surfaces the new DKB," which is the right tradeoff for the offline-first promise.
- **Step 12:** The pipeline runs in the browser only — Node-environment tests cover unit behavior but not the live flow. Open `npm run dev` and drop a real .pdf / .docx onto the marketing page to smoke-test. The classifier inference path currently passes an empty vocab map (`{ vocab: {} }`) plus the DKB-shipped patterns — once Step 11's `dkb-classifier-vocab.json` lands with real weights, change `extractAll`'s classifier options to read `dkb.classifier.vocab` directly. The `ensurePlaybooks` fetcher loops over `LAUNCH_PLAYBOOK_IDS` in order; this is sequential by design (the playbooks are tiny and a parallel `Promise.all` adds no measurable latency) but could be flipped if profiling shows otherwise. The dev middleware in `vite.config.ts` reads file contents per request without any caching — fine for dev, but if local DKB sizes balloon it should switch to `fs.readFileSync` on init + an in-memory map. Production headers (`_headers` CSP) are Step 14's concern; the dev middleware sets only `Content-Type`.
- **Step 19:** The starter DKB (`dkb/dist/v0.0.1-starter/`) is unchanged — v3 nodes live in their own files (`dkb/fixtures/v3/nodes/`) and are validated through `V3DkbNodeListSchema`. The v2 manifest does not yet reference them; Steps 21–22 will publish them under `dkb/dist/<version>/v3/`. The `dkb_node_version` integer is bumped only on breaking changes to a node's contract — additive optional fields can land without a bump.
- **Step 26:** The transfer ruleset uses three rule-scope configs (`CONFIG_SCC`, `CONFIG_UK`, `CONFIG_ALL`) to keep SCC-specific clauses from firing under the UK Addendum playbook (where the SCCs are incorporated by reference) and vice versa. This is the first ruleset to split scope by playbook within a single category — subsequent rulesets that bridge multiple regulators should follow the same pattern. The DPF detector (TRANSFER-017) is a warning per spec §31 because the EU-US Data Privacy Framework is the operative adequacy decision but permanently under litigation; downgrading to fail would gate normal commerce. The "modified SCCs" detector (TRANSFER-003) is intentionally aggressive — it matches "as modified", "as amended", "notwithstanding the SCCs", "with the exception" within 80 chars of "SCC"/"Standard Contractual Clauses". False positives are possible but the surface cost is low and Clause 2 is unforgiving. TRANSFER-016 was widened to accept either IDTA Part 1 *or* UK Addendum Table 1 so the same rule serves both UK transfer modes. TRANSFER-020 (onward transfers) is warning-level because the UK Addendum incorporates Clause 8.8 by reference.
- **Step 25:** USDPA-020 (claimed-Service-Provider-without-required-elements) uses a negative-lookahead regex: it fires when the document declares Service Provider status but does not contain "specific business purpose" or "same level of privacy protection" within 500 chars. The 500-char window is conservative enough to catch most patterns; if false-positives accumulate on real fixtures the window can be widened. USDPA-021 (multi-state strictness floor) is info-severity by design — the rule surfaces the risk without flagging a fail, because the actual strictest-floor analysis requires per-state requirement comparison that will land alongside Step 25's full compliance-matrix renderer. The seven state-statute snapshot fixtures from Step 21 already validate the rule citations; per-state rule expansion (e.g., a USDPA-VA-001 family) can land later as fixtures arrive.
- **Step 24:** The 55 rules use a shared generic factory at `src/engine/rules/v3/_regulated-rule.ts` rather than the BAA-specific `baa/_helpers.ts`. BAA's helpers still exist (no behavior change) but Step 25's US-state ruleset and Step 26's transfer ruleset will adopt the generic factory directly. SCC-Module-2 currently re-runs the full DPA-GDPR ruleset — the spec §41 SCC-Module-2-specific "exact-text comparison of SCC clauses" check is deferred to alongside the EU SCC `regulator_model_form` Annex-level extension (Step 22 followup note). The compliance-matrix columns on `dpa-controller-processor.json` are anchored to Article 28(3)(a)–(h) plus Articles 28(2), 32, 33(2), and Chapter V — the column mapping to rule ids will be wired by Step 32's report renderer. The `applies_to_playbooks` filter is verified by the inert-when-not-DPA test, so the v2 launch-suite `result_hash` is unchanged.
- **Step 23:** The 45 rules live in a single `rules.ts` instead of 45 individual files. Spec §6 step-6 note already flagged that the v2 scaffold puts each rule directly in `src/engine/rules/<category>/<RULE-ID>.ts` rather than the spec's per-rule-folder convention; the BAA ruleset takes one further step toward compactness because all 45 share the presence/language helper, making per-file copies pure boilerplate. Each rule's `id` (BAA-NNN), `citation`, `present_patterns` / `bad_patterns`, and recommendation text are individually reviewable in `rules.ts`. The detection regexes are intentionally conservative; real-world precision/recall calibration happens when the BAA fixture corpus from §15 lands (Step 34 golden tests). The pipeline integration appends `V3_RULES` after `LAUNCH_RULES` — the runner sorts lexicographically by id at execution time, so BAA-NNN rules cluster together in the execution log irrespective of import order. The CI matrix test (`test-matrix.yml`) uses the same `npm test` invocation and runs across Ubuntu/macOS/Windows — adding v3 BAA rules touches no OS-specific code so cross-OS determinism is unchanged for the v2 hash boundary, and the v3 determinism check in `baa-ruleset.test.ts` is the start of an analogous v3 hash boundary that Step 34 will extend.
- **Step 22:** The EU SCC parser detects all four Modules by regex but the spec calls for full per-clause parsing of Annexes I/II/III; the current implementation emits one `regulator_model_form` node per module with the three annexes as clause entries. When Step 24's DPA-GDPR ruleset lands, the SCC-Module-2 playbook (spec §41) will need annex-level clause access; that's a straightforward extension to the `ANNEX_CLAUSES` array. The EDPB index parser is intentionally light (regex over "Guidelines NN/YYYY") — spec §6 explicitly says "citation to document, not paragraph" because EDPB documents are not stable enough for paragraph-level pinning. The PIPL translation provenance is recorded as the `authority` field on every emitted node ("National People's Congress official English translation, 2021"); switching translations later is one line in `INTL_SOURCES.pipl.authority_label`. The Swiss Addendum URL (`edoeb.admin.ch`) is the FDPIC's published landing page; if the FDPIC's URL changes, update `SWISS_ADDENDUM_URL` and re-run `write-snapshots.ts`. The 4 international fetchers (PIPEDA/LGPD/APPI/PIPL) all emit a single contract-requirement node by design; subdivide per Step 25's multi-state-style overlays if rule-level granularity is needed.
- **Step 21:** Parsers extract requirements by matching hand-keyed regexes against snapshot text rather than fully parsing every statute — that's the right level of fidelity given the spec's "one structured `statutory_clause_requirement` set per source" framing, and it keeps the gate deterministic. The snapshot fixtures are concise representative excerpts authored from each source's published text; they exist to exercise the parsers offline and to feed the Step-20 staleness gate, not to be the canonical statute corpus. When Step 22's network-capable orchestrator lands, `write-snapshots.ts` can be replaced with a live fetcher that calls each source's HTTP endpoint and refreshes the snapshots. The HHS OCR resolution summaries directory (`dkb/fixtures/v3/ocr-resolutions/`) is created but empty — populating with real OCR settlement summaries is tracked alongside Step 22. The 7 state-privacy fetchers all emit a single processor-contract node by design; subdividing each state's requirements into per-paragraph nodes is straightforward (add entries to the relevant `STATE_PRIVACY_SOURCES[state].requirements` array) and can land alongside Step 25's rule work.
- **Step 20:** The CLI (`dkb/build/v3/check-staleness.ts`) is wired but not yet attached to a `npm run` script — Step 21's fetcher work is the natural place to add `dkb:check-staleness` since that's when real authority text first lands. The offline-mode snapshot scheme (`dkb/fixtures/v3/snapshots/{sha256(url)}.txt`) is content-addressed by `source_url` so the same key works for in-pipeline fetches and replay tests. The UI footer fetches `/dkb/v3/validation-status.json`; until Steps 21–22 land, that file is absent and the footer keeps its `—` / `0` placeholders, which is the right state for a brand-new repo. The build pipeline gate (non-zero exit on unacknowledged drift) is enforced at the `runStalenessCheck`/CLI layer only — wiring it into `build.ts` is one line and lands in Step 21 alongside the first real fetcher.
- **Step 11:** The orchestrator (`build.ts`) is end-to-end wired but has not been executed in this environment because every fetcher needs network access — the CI run is the verification gate.
