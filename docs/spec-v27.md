> **Status:** **Shipped (9.24.0).** v24 gave the posture archive a _per-front crossing-count_ axis (`crossings`); v25 re-bucketed those crossings _per step_ (`crossing_fronts`); v26 read the **index of the last one** ŌĆö the `settling_round` and whether the package was still moving at the close (`unsettled`). v26 reads the **latest** crossing ŌĆö and the latest throws away _when the first crossing happened_. The mirror axis a deal lead asks _opening_ a close, which v26 (the _last_-crossing index) structurally cannot pose: per deal, **how many rounds did the package hold the floor before its first slip ŌĆö did it start clean, or degrade from the very opening?** A deal whose crossings were a front falling in round 1ŌåÆ2 and again in the final round degraded _from the opening_ ŌĆö no grace period; a deal whose crossings were a front falling in round 4ŌåÆ5 and again in the final round is **identical to v26** (same `settling_round`, same `unsettled`) yet held a clean lead-in of three rounds before its first slip. It gates, with no tuning, on a deal that degraded from the opening. It continues the global step numbering after v26's Step 206, beginning at **Step 207**. One new subcommand, one pure module, **zero** change to any existing source file.
> **Scope:** one idea ŌĆö the _time-of-first-movement_ reduction the per-front crossing count (v24), the per-step crossing count (v25), and the _last_-crossing index (v26) all leave out. v24 answers _how many times did a front cross the floor_ (a per-front sum); v25 answers _how many fronts crossed together in each step_ (a per-transition count); v26 answers _which round did the package last cross the floor_ (the **max** index); v27 answers _which round did the package first cross the floor, and did it start clean_ (the **min** index). `total_crossings` equals v24's grand total, v25's per-step sum, and v26's per-step sum by construction ŌĆö v27 reads the same crossings for _where the first one falls_, not _how many there are_ or _where the last one falls_. # Part 0 ŌĆö Intent

## ┬¦1. Why we're doing this

v24 gave the posture archive a per-front crossing-count axis ŌĆö per front, how many times its standing crossed the acceptable floor (`crossings`), falls and recoveries alike. v25 re-bucketed those crossings by step (`crossing_fronts`). v26 located the **last** one ŌĆö the `settling_round`, the round the package last crossed the floor, gating on a deal still moving at the close. All three either reduce the crossings to a count or read the _latest_ of them. None can say _when the package first crossed the floor_. A deal lead opening a close asks a question the last-crossing index cannot answer: **how long did the package hold the floor before its first slip ŌĆö did the deal start clean, or degrade from the very opening?**

v26 cannot answer it, because it reads the **maximum** crossing index ŌĆö the last move, by construction silent on the first. Two deals v26 holds as identical can be opposites on this axis:

- **The early onset.** A front falls below floor in round 1ŌåÆ2 _and_ again in the final round. The package degraded _from the opening_: there was no grace period, it was exposed from the first exchange.
- **The clean lead-in.** A front falls below floor in round 4ŌåÆ5 _and_ again in the final round. To v26 this is the same as the early onset (same `settling_round` ŌĆö the final step; same `unsettled` ŌĆö yes; same `total_crossings`). Yet it held a **clean lead-in** of three steady rounds before its first slip ŌĆö exactly the difference a deal lead wants surfaced.

To v26 these are indistinguishable (same last crossing, same unsettled verdict). To v24 they may agree (each front with the same crossing count). None of the existing commands reports the one fact that separates a deal that degraded from the opening from one that held a clean lead-in: the **round the package first crossed the floor**, and whether that round was the opening.

This is not a bug in v26 ŌĆö a _settling_ index is supposed to read the last move, not the first. It is a missing **axis**: the _time-of-first-movement_ dimension of the same crossing data. Where v24 sums crossings down the front axis, v25 sums them down the step axis, and v26 reads the **maximum** crossing index, v27 reads the **minimum** crossing index. The first and last indices are independent whenever a deal crosses the floor more than once; they coincide only when a deal crosses exactly once (then onset = settling).

v27 supplies it. Read the same floor crossings v24/v25/v26 count for the _position_ of the first one ŌĆö and report the deal's **onset round** (the earliest round-transition any front crossed the floor), the **clean lead-in** (the run of steady rounds before it), and whether the **first** transition itself crossed (`early_onset`). The gate fires on an early onset ŌĆö the one time condition whose meaning ("the package was already moving across the floor at the opening") needs no tuning. All of this is a reduction of the crossings derivable from the `floors[]` array v22/v24/v25/v26 already carry ŌĆö the sum of every transition's crossings still equals v24's grand crossing total.

## ┬¦2. What v27 is and is not

**It is:**

- A **document-free onset command.** `coherence-onset <r1.coherence.json> ŌĆ” <rN.coherence.json>` reads N Ōēź 2 saved coherence artifacts in round order, verifies each, runs the cross-ladder guard across all of them, and prints the deal's **onset round** (`onset_round` ŌĆö the `to_round` of the earliest transition any front crossed the floor, `null` when none ever did), the **clean lead-in** (`lead_in` ŌĆö the run of steady steps before the first crossing), the **active-step count** (`active_count` ŌĆö steps where any front crossed), the **grand crossing total** (`total_crossings`, equal to v24's per-front sum and v25/v26's per-step sum), and a per-transition series marking each step `active` (a front crossed) or `still` (none did). Markdown (default) or `--format json`. No documents are read; the engine is never run.
- A **single deal-level onset gate.** `--fail-on-early-onset-exposure` exits 2 when the **first** transition crossed the floor (`early_onset`) ŌĆö the package was already moving across the floor at the opening. This is the _time-of-first-movement_ counterpart to v24's _per-front instability_ gate, v25's _per-step co-movement_ gate, and v26's _still-moving-at-the-close_ gate: where `--fail-on-unsettled-exposure` fires on the _last_ crossing falling on the _final_ step, `--fail-on-early-onset-exposure` fires on the single fact that the _first_ crossing fell on the _opening_ step. A deal that held a clean lead-in and then crossed late clears it (even if it ends unsettled); a deal that crossed in the opening round trips it (even though no front is volatile and no step synchronized). It needs no tuning ŌĆö a crossing in the opening round _is_ an early onset.
- A **whole-sequence-verified input, sharing the v17ŌĆōv26 front end.** Every artifact's `coherence_hash` is re-derived and checked on load (a tampered/corrupt round is a hard error, errors prefixed `round N:`). The v15/v16 cross-ladder guard runs across the sequence via the same `verifyCoherenceSequence` loader the ten trend/exposure/persistence/breadth/recurrence/volatility/synchrony/settling commands use, with no change to it.
- A **derived report carrying its own fingerprint.** The onset carries an `onset_hash` (a stable SHA-256 over the canonical per-transition set), namespaced apart from every `coherence_hash`, `movement_hash`, `trajectory_hash`, `shift_trajectory_hash`, `arc_hash`, `exposure_hash`, `persistence_hash`, `breadth_hash`, `recurrence_hash`, `volatility_hash`, `synchrony_hash`, and `settling_hash`, so the same N artifacts in the same order always reproduce the same report.

**It is not:**

- **Not a new diff, classifier, predicate, or artifact format.** It reuses the `weakest_tier` binding floor v12 derives and the `below-acceptable` rung v10 defines; v27 is a subcommand plus one pure module (`computeCoherenceOnset`) whose only new logic is locating the earliest floor crossing over the same `floors[]` array v24/v25/v26 scan. `parsePostureCoherenceJson`, the coherence schema, and every trajectory/exposure/persistence/breadth/recurrence/volatility/synchrony/settling function are unchanged.
- **Not a crossing-count command.** v24 counts a front's crossings over the deal; v25 counts the fronts crossing in each step. v27 does not headline a count ŌĆö it headlines an **index** (the round of the first crossing). Their totals agree (`total_crossings`).
- **Not the settling command.** v26 reads the **last** crossing (the `settling_round`, "had it stopped moving?"); v27 reads the **first** crossing (the `onset_round`, "did it start clean?"). They are mirror reductions of the same crossings ŌĆö `total_crossings` agrees ŌĆö and they coincide only when a deal crosses the floor exactly once. The moment a deal crosses twice, the first index and the last index are independent: v27 separates two deals v26 reports identically (same last crossing, different first). A consumer wanting the last crossing runs v26; one wanting the first runs v27; one wanting both runs both and joins on the shared `floors[]` matrix.
- **Not a duration command.** v21 sums how many rounds each front sat _below_ the floor (a length); v27 locates the round of the first _crossing_ (a position). A front stuck below floor for all N rounds (stated below from round 1) has a large v21 `rounds_below` but **zero** crossings ŌĆö it never moved across the floor in-sequence ŌĆö so it has no onset to v27 (`onset_round` is `null`). v27 reports _when movement started_, not _how long exposure lasted_.
- **Not silence-makes-an-opening-early.** A round no document states does **not** count as a crossing and does **not** reset the standing: per ┬¦3 (the v21/v24/v25/v26 contract), silence keeps the last known standing ŌĆö it is neither a recovery nor a fall. A crossing across a silent gap is attributed to the transition _into the round that reveals the new standing_, never to the silent step. So an opening round left entirely unstated reveals no crossing and leaves the lead-in honest (silence at the open is not an onset). This keeps v27 from inventing ŌĆö or mislocating ŌĆö a crossing at the opening.
- **Not a configurable "clean by round N" gate.** v27 does not take a `--clean-until N` knob; the gate fires on a crossing in the first transition, the one time condition whose meaning ("already moving at the opening") needs no tuning. A team wanting to require a _longer_ clean lead-in reads `lead_in` and `onset_round` from the JSON. (v20ŌĆōv26 made the same call against a configurable floor/duration/breadth/episode/crossing count/simultaneity/quiet tail.)
- **Not a trajectory-whipsaw command.** v17's `whipsaw` fires on rung jitter anywhere on the ladder, including movement entirely _above_ the floor. v27's onset is specific to the **floor boundary**: an above-floor whipsaw (`acceptable ŌåÆ ideal ŌåÆ acceptable`) crosses the floor zero times, so it has no onset round (`null`) and is not an early onset.
- **Not an onset artifact.** v27 introduces no new on-disk format; the onset is _derived_, recomputed on demand from the N auditable coherence inputs ŌĆö the same "keep the derived thing derived" discipline v14ŌĆōv26 hold.
- **Not a sequence the command discovers.** It does not glob a directory or sort filenames; the caller passes the artifacts **in round order** on the argv, exactly as the ten trend/exposure/persistence/breadth/recurrence/volatility/synchrony/settling commands do.
- **Not a browser surface.** An N-round onset of uploaded artifacts is a CI/dashboard concern, which is what this command serves (mirroring v16ŌĆōv26).

## ┬¦3. The posture filter (unchanged)

1. **Deterministic** ŌĆö `computeCoherenceOnset` over N `PostureCoherence` objects is pure: it matches fronts by dimension, pins each step's crossed-dimension list by `localeCompare(_, "en")` (the same order the trajectory/exposure/volatility/synchrony/settling functions use), and locates the earliest floor crossing over the caller's round order. Identical artifacts in identical order ŌåÆ identical `onset_hash` on any machine.
2. **Honest about unstated data** ŌĆö a front no document states in a round does not cross into or out of that round. Within a front's path, an unstated round is skipped: it neither counts as a crossing nor resets the standing (silence is not a fall or a recovery ŌĆö the ┬¦3 contract that keeps `unstated` never-flagged in v20 and `below ŌåÆ unstated ŌåÆ below` zero crossings in v24/v25/v26). A crossing across a silent gap is attributed to the transition into the round that _reveals_ the new standing; an opening round left unstated reveals no crossing, so the lead-in stays honest.
3. **Advisory** ŌĆö the command names the round the team's own floor was first crossed. It asserts no legal conclusion.
4. **No server** ŌĆö N local files in, one report out. No socket, no engine run.
5. **Additive** ŌĆö a brand-new subcommand and one new pure module. Every existing command (`analyze`, `diff`, `compare`, `compare-coherence`, `coherence-trend`, `coherence-shift-trend`, `coherence-arc`, `coherence-exposure`, `coherence-persistence`, `coherence-breadth`, `coherence-recurrence`, `coherence-volatility`, `coherence-synchrony`, `coherence-settling`, `verify`) is byte-for-byte unchanged in _output and goldens_; `coherence-sequence.ts` is imported and reused **without modification** (v27 needs nothing exported that was not already public ŌĆö like v19ŌĆōv26, it touches no existing source file's behavior at all).

# Part I ŌĆö The onset (pure)

`src/report/coherence-onset.ts` (a new sibling to `coherence-settling.ts`):

- **`OnsetState`** ŌĆö a transition's onset state: `active` (at least one front crossed the floor in this step), `still` (none did).
- **`CoherenceOnsetTransition`** ŌĆö one round-transition's movement read for _when_ the first crossing falls: the `from_round` / `to_round` (1-based), `crossing_fronts` (how many fronts crossed), `crossed_dimensions` (the fronts that crossed, `localeCompare`-pinned), and the `state`.
- **`CoherenceOnset`** ŌĆö the whole-sequence reduction: `rounds` (count), `per_transition[]` (the transitions above, in sequence order, length = rounds ŌłÆ 1), `total_crossings` (every front's crossing summed over the steps ŌĆö equal to v24's, v25's, and v26's total), `onset_round` (the `to_round` of the earliest `active` step ŌĆö the round the package first crossed the floor; `null` when none ever crossed), `lead_in` (the run of leading `still` steps before the first crossing ŌĆö the whole sequence when none crossed), `active_count` (steps where any front crossed), `early_onset` (the first transition was `active`), and `onset_hash`.
- **`computeCoherenceOnset(rounds: PostureCoherence[])`** ŌĆö the pure, IO-free core. For each front, scans `floors[]` and attributes each _stated_ floor crossing to the transition that reveals it (an unstated round is skipped, ┬¦3); marks each transition `active`/`still`, locates the earliest active step, computes the clean lead-in, and returns the reduction with an `onset_hash`. Requires Ōēź 2 rounds.
- **`exposureEarlyOnset(onset)`** ŌĆö the CI gate predicate: `onset.early_onset`. The time-of-first-movement verdict no other command exposes (v24 gates on a single front's crossing count; v25 on fronts crossing together in one step; v26 on the last crossing falling on the final step; this gates on the _first_ crossing falling on the _opening_ step).
- **`buildCoherenceOnsetJson(onset)`** / **`renderCoherenceOnsetSummary(onset)`** ŌĆö the JSON (`schema: vaulytica.posture-onset.v1`) and human-readable renderers. The summary prints the onset verdict (at round K with a clean lead-in, or EARLY at the opening round, or steady-throughout when nothing crossed), the active-step count, then one line per step, then the hash.

v27 imports only the already-public `PostureCoherence` type, the `NegotiationTier` type, and the shared hashing helpers; the `below-acceptable` literal is a plain string; no existing source file changes.

# Part II ŌĆö The command (headless)

`tools/cli/coherence-onset.ts` (a new sibling to `coherence-settling.ts`):

- **`computeCoherenceOnsetArtifacts(texts, format?)`** ŌĆö the pure CLI core. # Part XVIII ŌĆö What this gives the user

- **The onset of the deal, from the archive alone.** Archive each round's kilobyte coherence artifact as you go; `coherence-onset` shows not just _how many times a front crossed the floor over the deal_ (v24), _how many fronts crossed together in each round_ (v25), or _which round the package last crossed the floor_ (v26) but _which round the package first crossed the floor, and whether it started clean_ ŌĆö distinguishing an early onset that degraded from the opening from a clean lead-in that held the floor for several rounds, which v26 reports identically (same last crossing, same unsettled verdict), with no clause text checked out and no re-analysis run. The "did the package start clean?" question a deal lead asks opening a close, answered deterministically from N small, verifiable files.
- **A gate on the time of the first move, not just the crossing count, level, or last move.** `--fail-on-early-onset-exposure` fires when the _first_ transition crossed the floor ŌĆö the package was already in motion at the opening ŌĆö a condition the count gates (v23/v24/v25) and the last-move gate (v26) structurally cannot pose.
